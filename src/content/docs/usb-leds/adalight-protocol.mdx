LumaSync's USB pipeline speaks the **Adalight protocol**: a simple
byte-oriented frame format over a USB serial link. This page explains what
that means in practice — useful if you're writing your own firmware or
debugging why a kit isn't lighting up.

## Why Adalight

Adalight was originally a 2011 Arduino project from Adafruit for building
DIY bias lighting. Its protocol was trivial: a magic header, a length word,
and a stream of RGB triplets. That simplicity meant every Arduino-based
ambilight kit since has adopted it, either directly or with minor
extensions.

LumaSync targets unmodified Adalight firmware. If your controller lights up
from Prismatik, Hyperion, or a standalone Adalight host, it will light up
from LumaSync.

## Wire format

Each frame is:

```
Header:  'A' 'd' 'a'           (3 bytes, ASCII magic)
         LED_COUNT_HI            (1 byte, high byte of LED count - 1)
         LED_COUNT_LO            (1 byte, low byte of LED count - 1)
         LED_COUNT_HI ^ LED_COUNT_LO ^ 0x55   (1 byte, checksum)
Payload: RGB triplets             (LED_COUNT * 3 bytes, one per LED)
```

For a 120-LED strip that's 6 bytes of header + 360 bytes of colour data =
366 bytes per frame. At 20 Hz that's 7.3 KB/s — well within 115200 baud's
~11.5 KB/s budget.

## Baud rate

**115200**. Hard-coded in LumaSync. Most Adalight firmwares ship at this
rate; if yours doesn't, reflash it. Raising the baud rate doesn't help —
the strip itself is the bottleneck.

## The handshake

When you click **Health check** in Settings → Device, LumaSync:

1. Opens the serial port at 115200-8-N-1.
2. Waits 2 seconds for the controller to finish its reset (Arduino-class
   boards auto-reset on serial open).
3. Sends a minimal zero-LED-count Adalight frame — magic header + length
   word + correct checksum + no payload.
4. Looks for an ACK byte or just a quiet serial line with no errors on the
   port. Most Adalight firmwares don't send a response; silence after the
   header is treated as "alive".
5. Reports `DEVICE_READY` in the health check result.

If the port opens but the handshake fails, LumaSync surfaces it as
`DEVICE_MANUAL_REQUIRED` — connect but don't assume handshake success.

## Frame rate

LumaSync pushes 30 frames per second on the USB path by default. The serial
link can handle it for strips up to ~300 LEDs; longer strips need a lower
rate to avoid buffering. The telemetry pill on the Device panel shows the
actual push rate.

## Extensions

LumaSync does **not** implement Adalight extensions:

- No brightness-byte envelope
- No DPX / BGRW payload variants
- No per-LED white channel (the strip is assumed RGB, not RGBW)

If your firmware expects any of these, configure it to vanilla Adalight
mode first. Most firmwares default to vanilla.

## Why not raw APA102 / SPI?

Adalight-over-USB-serial buys hot-pluggable hardware at the cost of a
latency ceiling. SPI-attached strips (APA102, SK9822) are faster and don't
need a microcontroller intermediary — but they require wiring into the host
machine, which excludes 99% of desktop users. USB-serial is the pragmatic
path.
